# Repo-rter v0.4.3 — 기술 부채 청소 및 데이터 정합성 검증 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 초기 커밋부터 동작하지 않던 백그라운드 동기화를 고치고, 데이터 병합 로직에 테스트 안전망을 씌우고, Tauri RC/stable 버전 불일치와 742줄짜리 i18n 하드코딩을 정리한다.

**Architecture:** 새 기능은 없다. 전부 기존 코드의 결함 수정과 구조 정리다. 테스트 인프라(vitest)를 먼저 세우고, 그 위에서 버그를 red-green으로 잡은 뒤, 검증된 상태에서 의존성 버전과 파일 구조를 건드린다. 각 태스크는 독립적으로 되돌릴 수 있고 순서대로 실행하면 항상 빌드가 통과한다.

**Tech Stack:** Next.js 16 (static export) / React 19 / TypeScript 5 / Tauri 2 (Rust) / vitest 4 + jsdom / i18next

## Global Constraints

프로젝트 전역 요구사항. 모든 태스크의 요구사항에 암묵적으로 포함된다.

- **키체인/OS 보안 저장소는 이번 릴리스 범위 밖이다.** `tauri-plugin-secure-storage` 도입은 v0.4.4로 분리됐다. 이번 릴리스에서 `src/lib/auth.ts`의 localStorage 저장 방식은 **건드리지 않는다.** (배경: 아래 "0.4.4로 이월된 항목" 참조)
- **Rust 크레이트 버전은 아래 값을 정확히 사용한다** (crates.io 2026-07-17 기준 최신 stable, 추측 금지):
  - `tauri = "2.11.5"` / `tauri-build = "2.6.3"`
  - `tauri-plugin-http = "2.5.9"` / `tauri-plugin-shell = "2.3.5"` / `tauri-plugin-notification = "2.3.3"` / `tauri-plugin-log = "2.9.0"`
  - `rust-version`은 `"1.77.2"` 이상 (tauri 2.11.5의 MSRV)
- **JS 측 Tauri 패키지는 이미 stable이며 위 Rust 버전과 짝이 맞는다.** `@tauri-apps/plugin-http@^2.5.9`, `plugin-notification@^2.3.3`, `plugin-shell@^2.3.5`. 이번 작업은 **Rust를 JS에 맞추는 것**이지 그 반대가 아니다. JS 플러그인 버전은 올리지 않는다.
- **저장된 사용자 데이터 형식은 절대 변경 금지.** 캐시 파일은 `{ timestamp, count, uniques }` 객체의 JSON 배열이고, 병합 키는 `timestamp`의 앞 10자(`YYYY-MM-DD`)다. 기존 사용자의 히스토리는 GitHub에서 재취득이 불가능하다(14일 초과분은 영구 소실).
- **번역 키는 12개 언어 × 56개로 완전히 동일해야 한다.** 언어 코드: `en ko zh es hi ar fr ru pt de ja id`. 키를 추가·삭제·번역하지 않는다. 이번 작업은 순수한 파일 이동이다.
- 커밋 메시지는 Conventional Commits(`feat:`, `fix:`, `test:`, `refactor:`, `chore:`)를 따른다.

```ts
import { describe, it, expect, beforeEach, vi } from 'vitest';

vi.mock('@tauri-apps/api/core', () => ({
  isTauri: () => false,
  invoke: vi.fn(),
}));

import { mergeTrafficData } from '@/lib/storage';

const point = (timestamp: string, count: number) => ({ timestamp, count, uniques: 1 });

describe('mergeTrafficData (브라우저 localStorage 경로)', () => {
  beforeEach(() => {
    localStorage.clear();
  });

  it('저장된 데이터가 없으면 incoming을 그대로 반환하고 저장한다', async () => {
    const incoming = [point('2026-06-01T00:00:00Z', 10)];

    const merged = await mergeTrafficData('owner/repo', 'views', incoming);

    expect(merged).toEqual(incoming);
    expect(JSON.parse(localStorage.getItem('gittraffic_views_owner/repo')!)).toEqual(incoming);
  });

  it('GitHub 14일 창을 벗어난 과거 데이터를 보존한다', async () => {
    // 앱의 존재 이유: GitHub 응답에서 사라진 과거 데이터가 살아남아야 한다.
    const old = [point('2026-01-01T00:00:00Z', 1), point('2026-02-01T00:00:00Z', 2)];
    localStorage.setItem('gittraffic_views_owner/repo', JSON.stringify(old));

    const merged = await mergeTrafficData('owner/repo', 'views', [point('2026-07-01T00:00:00Z', 3)]);

    expect(merged).toHaveLength(3);
    expect(merged.map((p) => p.timestamp)).toEqual([
      '2026-01-01T00:00:00Z',
      '2026-02-01T00:00:00Z',
      '2026-07-01T00:00:00Z',
    ]);
  });

  it('같은 날짜는 incoming 값으로 덮어쓰고 중복을 만들지 않는다', async () => {
    localStorage.setItem('gittraffic_views_owner/repo', JSON.stringify([point('2026-06-01T00:00:00Z', 10)]));

    const merged = await mergeTrafficData('owner/repo', 'views', [point('2026-06-01T00:00:00Z', 42)]);

    expect(merged).toHaveLength(1);
    expect(merged[0].count).toBe(42);
  });

  it('입력 순서와 무관하게 시간순으로 정렬한다', async () => {
    localStorage.setItem('gittraffic_views_owner/repo', JSON.stringify([point('2026-06-03T00:00:00Z', 3)]));

    const merged = await mergeTrafficData('owner/repo', 'views', [
      point('2026-06-05T00:00:00Z', 5),
      point('2026-06-01T00:00:00Z', 1),
    ]);

    expect(merged.map((p) => p.timestamp)).toEqual([
      '2026-06-01T00:00:00Z',
      '2026-06-03T00:00:00Z',
      '2026-06-05T00:00:00Z',
    ]);
  });

  it('views와 clones를 서로 다른 키에 저장한다', async () => {
    await mergeTrafficData('owner/repo', 'views', [point('2026-06-01T00:00:00Z', 10)]);
    await mergeTrafficData('owner/repo', 'clones', [point('2026-06-01T00:00:00Z', 99)]);

    expect(JSON.parse(localStorage.getItem('gittraffic_views_owner/repo')!)[0].count).toBe(10);
    expect(JSON.parse(localStorage.getItem('gittraffic_clones_owner/repo')!)[0].count).toBe(99);
  });
});
```

### Task 2: 백그라운드 동기화가 한 번도 동작하지 않던 버그 수정

**이번 릴리스에서 사용자 가치가 가장 큰 변경이다.**

`src/hooks/useBackgroundSync.ts:35`가 `localStorage.getItem('github_token')`을 읽고 `if (!token) return;`으로 빠져나간다. 그런데 `github_token`을 **쓰는 코드는 저장소 전 히스토리의 어떤 ref에도 존재한 적이 없다.** 토큰을 저장하는 곳은 `src/lib/auth.ts:5`이고 키는 `github_pat`이다.

근본 원인은 오타가 아니라 **훅이 auth 모듈을 우회해 저장 키를 직접 하드코딩한 것**이다. 그래서 `'github_pat'`으로 문자열만 바꾸지 않고 `getGithubToken()`에 위임해 단일 진실 공급원을 만든다. Task 6(v0.4.4)에서 auth가 async 키체인으로 바뀌어도 이 지점은 자동으로 따라온다.

### Task 3: Rust `merge_lists` 유닛 테스트

Tauri 경로의 병합은 `src-tauri/src/lib.rs:116`의 `merge_lists`가 담당한다. Task 1이 TS 구현을 덮었으니 여기서 Rust 구현을 덮는다. 데스크톱 앱의 실사용 경로가 이쪽이므로 오히려 더 중요하다.

`merge_lists`는 비공개 함수지만 같은 크레이트 안의 `#[cfg(test)] mod tests`에서 `super::merge_lists`로 접근 가능하다. **가시성을 바꾸지 말 것.**

**Files:**
- Modify: `src-tauri/src/lib.rs` (파일 맨 끝에 테스트 모듈 추가)

**Interfaces:**
- Consumes: `fn merge_lists(existing: &Vec<serde_json::Value>, incoming: &Vec<serde_json::Value>) -> Vec<serde_json::Value>` — `src-tauri/src/lib.rs:116`
- Produces: `cargo test`로 실행되는 테스트 모듈. 다른 태스크가 의존하지 않는다.

- [ ] **Step 1: 테스트 모듈 작성**

`src-tauri/src/lib.rs` 맨 끝(`pub fn run()` 블록 다음)에 추가한다.

```rust
#[cfg(test)]
mod tests {
    use super::merge_lists;
    use serde_json::json;

    fn point(timestamp: &str, count: u64) -> serde_json::Value {
```

```js
import fs from 'node:fs';
import path from 'node:path';

const src = fs.readFileSync('src/lib/i18n.ts', 'utf8');
const start = src.indexOf('const resources =');
const end = src.indexOf('i18n\n  .use');
if (start === -1 || end === -1) throw new Error('i18n.ts의 resources 블록을 찾지 못했습니다');

const body = src.slice(start + 'const resources ='.length, end).trim().replace(/;$/, '');
const resources = eval('(' + body + ')');

const dir = 'src/lib/locales';
fs.mkdirSync(dir, { recursive: true });

const langs = Object.keys(resources);
for (const lang of langs) {
  const translation = resources[lang].translation;
  fs.writeFileSync(
    path.join(dir, `${lang}.ts`),
    `const ${lang} = ${JSON.stringify(translation, null, 2)};\n\nexport default ${lang};\n`
  );
}

const imports = langs.map((l) => `import ${l} from './${l}';`).join('\n');
const entries = langs.map((l) => `  ${l}: { translation: ${l} },`).join('\n');
fs.writeFileSync(
  path.join(dir, 'index.ts'),
  `${imports}

export const resources = {
${entries}
};

export default resources;
`
);

console.log(`${langs.length}개 언어 파일 생성: ${langs.join(', ')}`);
console.log(`키 개수: ${Object.keys(resources.en.translation).length}`);
```

## 이번 릴리스의 비목표 (Non-goals)

- **키체인/OS 보안 저장소** — v0.4.4로 이월. 아래 참조.

- **`Dashboard.tsx` 719줄 분리** — 별도 리팩터링 릴리스로.
- **`tauri.conf.json`의 `"csp": null`** — 보안 강화 작업이라 v0.4.4의 키체인 작업과 함께 다루는 것이 맞다.

## 0.4.4로 이월된 항목: OS 키체인 보안 저장소

원래 0.4.3 기획안의 핵심이었으나 아래 사실이 확인되어 분리했다. **0.4.4 기획 시 이 조사 결과에서 출발할 것.**

- **`@tauri-apps/plugin-secure-storage`는 npm에 존재하지 않는다 (404).** `@tauri-apps`는 Tauri 공식 조직 스코프이고 secure-storage는 공식 플러그인이 아니다. JS 바인딩이 없으므로 `setPassword`/`getPassword`/`deletePassword` 임포트 기반 설계는 성립하지 않는다.
- **Rust 크레이트 `tauri-plugin-secure-storage`는 존재하지만 최신이 1.5.0이다** (2.0.0 아님). Tauri v2 호환은 맞다 (`tauri ^2.6.2`, `keyring ^3` 의존). 다만 최근 다운로드 1,437회, crates.io상 저장소 URL 없음 — 감사되지 않은 개인 크레이트에 사용자 PAT를 맡기는 셈이다.
- **권장 대안: `keyring` 크레이트를 직접 의존성에 넣고 커맨드 3개를 작성한다.** 위 플러그인이 하는 일이 정확히 그 래핑이다. `lib.rs`에 이미 같은 패턴의 커맨드가 5개 있으니 30줄 남짓이면 되고, 없는 JS 바인딩 문제도 사라진다. (2026-07-17 기준 `keyring` 최신은 4.1.5 — 착수 시 재확인할 것)
- **Linux CI가 깨진다.** `keyring`은 리눅스에서 libsecret에 링크한다. `.github/workflows/release.yml`의 ubuntu-22.04 apt 단계에 `libsecret-1-dev`를 추가해야 하고, Secret Service 데몬이 없는 환경을 위한 런타임 폴백도 필요하다.
- **웹 빌드 경로를 설계에 넣어야 한다.** 이 앱은 `output: 'export'`이고 코드 전반이 `isTauri()`로 분기하는데 `src/lib/auth.ts`에는 그 가드가 없다. 키체인은 Tauri 안에만 있으므로 `npm run dev` 경로의 localStorage 폴백이 필요하다.
- **`sync_encryption_key`를 빠뜨리지 말 것.** 원 기획안은 PAT와 WebDAV 패스워드만 다뤘으나, E2EE 패스프레이즈가 업로드된 백업 전체를 여는 열쇠다. 셋 중 가장 민감하다. (`src/lib/sync.ts:29`, `:73`)
- **async 전환 비용은 작다.** `getGithubToken`의 실제 호출처는 `src/lib/github.ts:9`의 `getAuthHeaders` 하나뿐이고, `hasGithubToken`은 `src/app/page.tsx:12` 하나다. Task 2 이후 `useBackgroundSync`도 `getGithubToken()`을 경유하므로 자동으로 따라온다.
- **localStorage → 키체인 마이그레이션**과 기존 평문 토큰 삭제가 필요하다.
